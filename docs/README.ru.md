## 📖 Почему «Jandaira»?

**Jandaíra** (*Melipona subnitida*) — безжальная пчела, эндемичная для биома Каатинга в Бразилии. Маленькая, стойкая и исключительно кооперативная — она не нуждается в централизованном лидере, чтобы построить функциональный улей. Каждая рабочая пчела знает свою роль, автономно выполняет задачу и возвращает результат коллективу.

Именно эта архитектурная модель реализована в данном проекте:

- **Королева (`Queen`)** не выполняет задачи — она оркестрирует, проверяет политики и обеспечивает безопасность.
- **Специалисты (`Specialists`)** — лёгкие агенты с ограниченным набором инструментов, работающие в изолированных силосах.
- **Нектар (Nectar)** — метафора для бюджета токенов: каждый агент потребляет нектар; когда он заканчивается, улей останавливается.
- **Соты (`Honeycomb`)** — двухуровневая система персистентной памяти: `ShortTermMemory` хранит актуальный контекст в RAM с автоматическим TTL; встроенный `VectorEngine` (BadgerDB + HNSW) архивирует консолидированные знания в виде долгосрочных векторов — без внешних зависимостей.
- **Граф знаний (`KnowledgeGraph`)** — карта связей между агентами, темами и инструментами: Королева обращается к нему перед каждой миссией, чтобы переиспользовать профили специалистов, уже успешно справлявшихся с похожими задачами.
- **Пасечник (Beekeeper)** — человек в контуре: он может одобрить или заблокировать любое действие ИИ до его выполнения.
- **Вебхуки (Webhooks)** — внешние HTTP-триггеры, позволяющие системам CI/CD, GitHub, Prometheus и подобным запускать рои агентов простым вызовом URL. Поле **GoalTemplate** использует встроенный `text/template` Go для превращения входящего JSON-payload в цель, которую Королева обработает — без перекомпиляции бинарника.

## 🧠 Архитектура памяти

`internal/brain/` — это не просто векторное хранилище: пакет реализует двухуровневую иерархию памяти с графом знаний, который растёт с каждой миссией.

### Кратковременная память — `ShortTermMemory`

`brain/short_term.go` — буфер сообщений с TTL на каждую запись. Он решает проблему переполнения контекста LLM в долгих сессиях:

- При вставке каждое сообщение получает метку времени истечения
- Истёкшие записи тихо удаляются при следующем обращении
- **Автоматическое уплотнение**: когда буфер достигает `maxEntries`, LLM суммаризует накопленную историю в плотный абзац → итог встраивается и архивируется в VectorEngine как `short_term_archive` → RAM-буфер очищается
- `Flush(ctx)` следует вызывать в конце сессии для полной архивации; при сбое LLM сырой транскрипт архивируется как запасной вариант

```
 Вставка нового сообщения
         │
         ▼
┌──────────────────────────────────┐
│      ShortTermMemory (RAM)       │
│  [msg₁ · истекает: +30мин]      │
│  [msg₂ · истекает: +30мин]      │
│  ...                             │
│  [msgN · истекает: +30мин]      │ ← переполнение: compact() запускается
└──────────────────────────────────┘
         │
         ▼
   LLM суммаризует историю
         │
         ▼
┌──────────────────────────────────┐
│  VectorEngine (Долгосрочная)     │
│  type: "short_term_archive"      │
└──────────────────────────────────┘
```

### Граф знаний — `KnowledgeGraph` (GraphRAG)

`brain/graph.go` реализует граф знаний, персистированный в JSON (`~/.config/jandaira/knowledge_graph.json`), который автоматически накапливает экспертизу после каждого завершённого рабочего процесса.

**Модель данных**

| Элемент | Тип | Пример |
|---|---|---|
| Профиль специалиста | узел `agent` | `"Аналитик данных"` |
| Домен миссии | узел `topic` | `"анализ финансового отчёта"` |
| Связь экспертизы | ребро `expert_in` | `agent → topic` |

**Автоматический цикл обучения Королевы**

После каждого рабочего процесса `registerWorkflowInGraph` выполняет:
1. Создаёт/обновляет узел `topic` с целью миссии
2. Для каждого специалиста конвейера создаёт/обновляет узел `agent`
3. Создаёт рёбра `expert_in`, связывающие агентов с темой

Перед сборкой следующего роя `graphContextForGoal` запрашивает граф по ключевым словам новой цели и внедряет блок **"PAST SPECIALIST KNOWLEDGE"** в мета-планировочный промпт Королевы.

Результат: Королева проектирует всё более совершенные рои со временем, без дополнительных вызовов LLM.

### Почему Go превосходит Python здесь?

1. **Горутины дешевле потоков** — запуск 100 одновременных агентов стоит ничтожно мало по сравнению с Python `asyncio` или `threading`.
2. **Статический бинарный файл** — нулевые зависимости во время выполнения. `go build` генерирует исполняемый файл, который работает на любом Linux без установки чего-либо.
3. **Нет GIL** — в Python есть Global Interpreter Lock; Go по-настоящему параллелизует работу на нескольких ядрах.
4. **`wazero` — это 100% Go** — Wasm-среда выполнения не требует CGo, Docker или внешних систем. Агент работает в песочнице внутри того же процесса.

## 🚀 Руководство по использованию

### Предварительные требования

```bash
# Go 1.22 или выше
go version

# Ключ OpenAI API
export OPENAI_API_KEY="sk-..."
```

> **Docker не требуется.** Векторная база данных (`VectorEngine`) встроена в бинарный файл и автоматически сохраняет данные в `~/.config/jandaira/vectordb/`.

### Установка

#### Вариант 1 — Сборка из исходного кода

```bash
git clone https://github.com/damiaoterto/jandaira.git
cd jandaira

# Загрузить зависимости
go mod tidy

# Собрать API-сервер
go build -o jandaira-api ./cmd/api/
```

#### Вариант 2 — Запустить напрямую

```bash
go run ./cmd/api/main.go --port 8080
```

### Запуск улья

```bash
./jandaira-api --port 8080
```

Сервер будет доступен по адресу `http://localhost:8080`. Мониторинг событий в реальном времени через WebSocket: `ws://localhost:8080/ws`.

### Пример: создание и тестирование файла Go

1. Отправьте цель через `POST /api/dispatch`:

   ```bash
   curl -X POST http://localhost:8080/api/dispatch \
     -H "Content-Type: application/json" \
     -d '{"goal": "Создай файл Go с именем sum.go, который складывает два числа", "group_id": "enxame-alfa"}'
   ```

2. Королева распределяет задачу по конвейеру специалистов:
   - **Wasm-разработчик** → пишет `sum.go` с помощью `write_file`
   - **Аудитор качества** → выполняет код с помощью `execute_code` и генерирует отчёт

3. Отслеживайте прогресс через WebSocket:

   ```json
   { "type": "agent_change", "agent": "Wasm-разработчик" }
   { "type": "tool_start",   "agent": "Wasm-разработчик", "tool": "write_file", "args": "{...}" }
   { "type": "result",       "message": "# Итоговый отчёт\n..." }
   ```

4. Если `RequiresApproval: true`, **режим пасечника** активируется. Сервер отправляет `approval_request` через WebSocket и ждёт ответа:

   ```json
   // Сервер отправляет:
   { "type": "approval_request", "id": "req-1712345678901", "tool": "write_file", "args": "{...}" }

   // Клиент отвечает:
   { "type": "approve", "id": "req-1712345678901", "approved": true }
   ```

5. В конце результат встраивается и сохраняется в локальный VectorEngine для будущего использования.

### Настройка собственного роя

Отредактируйте `cmd/api/main.go`, чтобы определить политику роя:

```go
queen.RegisterSwarm("my-swarm", swarm.Policy{
    MaxNectar:        50000,  // Бюджет токенов
    Isolate:          true,   // Изолированный контекст на группу
    RequiresApproval: true,   // Режим пасечника (HIL)
})
```

### Доступные инструменты

| Инструмент | Описание |
|---|---|
| `list_directory` | Перечисляет файлы и папки в директории |
| `read_file` | Читает содержимое файла |
| `write_file` | Создаёт или перезаписывает файл |
| `execute_code` | Выполняет код в изолированной Wasm-песочнице |
| `web_search` | Поиск в интернете через DuckDuckGo (прямые ответы, определения, аннотации) |
| `search_memory` | Семантический поиск в векторной памяти улья (BadgerDB + HNSW) |
| `store_memory` | **Единственный механизм постоянного хранения.** Сохраняет данные во встроенный VectorEngine с необязательными полями `type` и `metadata`. |

## 🔐 Безопасность

Каждая «эстафетная передача» между специалистами **зашифрована с помощью AES-GCM**:

1. В начале каждого рабочего процесса генерируется эфемерный сессионный ключ
2. Накопленный контекст **шифруется перед отправкой** следующему специалисту
3. Специалист получает зашифрованную полезную нагрузку, расшифровывает, обрабатывает и **повторно шифрует** свой ответ
4. Никакой контекст не передаётся в открытом виде между агентами

Это имитирует защищённый IPC-канал, где даже если один агент скомпрометирован, он не может прочитать историю других агентов в конвейере.

## 🪝 Webhook Engine

**Webhook Engine** позволяет внешним системам активировать ульи по HTTP без дополнительной аутентификации. Каждый вебхук имеет уникальный `slug` в URL, **GoalTemplate** на основе встроенного `text/template` Go и опциональный `secret` для валидации HMAC-SHA256.

### Поток выполнения

```
Внешняя система (GitHub, Prometheus, Slack, CI/CD…)
     │  POST /api/webhooks/monitor-deploy
     │  Body: {"project_name": "Jandaira", "env": "prod"}
     ▼
┌─────────────────────────────────────────────┐
│              Webhook Engine                  │
│                                             │
│  1. Поиск вебхука по slug                   │
│  2. Валидация HMAC-SHA256 (если secret задан)│
│  3. Рендер GoalTemplate с payload:          │
│     "Проанализируй деплой Jandaira в prod"  │
│  4. Вызов Queen.DispatchWorkflow            │
└─────────────────────────────────────────────┘
     │
     ▼
Королева → AssembleSwarm / BuildSpecialists → Результат через WebSocket
```

### GoalTemplate

Использует встроенный `text/template` Go. Любое поле JSON-payload доступно через `{{.поле}}`:

```
"Проанализируй деплой проекта {{.project_name}} в среде {{.env}}"
"Алерт: {{.alertname}} — инстанс {{.instance}} ({{.severity}})"
"PR #{{.number}} в {{.repository.name}}: {{.title}}"
```

### Валидация HMAC-SHA256

Если `secret` задан, вызывающая сторона должна передать заголовок:

```
X-Hub-Signature-256: sha256=<hex-encoded-HMAC-SHA256-тела>
```

Совместимо со стандартом GitHub Webhooks. Payload с невалидной подписью получает `401 Unauthorized`.

### Outbound Webhooks (Исходящие вебхуки)

Jandaira также поддерживает **Outbound Webhooks**, позволяя улью автоматически отправлять результат своей работы во внешние системы (Discord, Slack и т.д.) сразу после завершения миссии. Формат запроса настраивается через `BodyTemplate`.

В шаблоне есть важные встроенные функции (фильтры) для отправки структурированных JSON payloads:
- `json`: Безопасно экранирует текст (переносы строк, кавычки) внутри JSON.
- `truncate <лимит>`: Обрезает строку до максимальной длины, предотвращая ошибки в API (например, лимит 2000 символов в Discord).
- `normalize`: Очищает текст ИИ, удаляя метаданные оркестрации, внутренние логи и дампы памяти, оставляя только финальный отчет последнего агента.

**Пример исходящего payload:**
```json
{
  "content": {{.result | normalize | truncate 1900 | json}}
}
```

### Полный пример

```bash
# Создать вебхук
curl -X POST http://localhost:8080/api/webhooks \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Monitor Deploy",
    "slug": "monitor-deploy",
    "colmeia_id": "<id-улья>",
    "secret": "мой-секрет",
    "goal_template": "Проанализируй деплой {{.project_name}} в среде {{.env}}",
    "active": true
  }'

# Запустить (с HMAC-подписью)
BODY='{"project_name":"Jandaira","env":"prod"}'
SIG="sha256=$(echo -n "$BODY" | openssl dgst -sha256 -hmac "мой-секрет" | awk '{print $2}')"

curl -X POST http://localhost:8080/api/webhooks/monitor-deploy \
  -H "Content-Type: application/json" \
  -H "X-Hub-Signature-256: $SIG" \
  -d "$BODY"
```

## 🌐 Справочник API

Запустите HTTP-сервер командой `./jandaira-api --port 8080`. Доступны следующие маршруты:

### REST-маршруты

#### Настройка и отправка

| Метод | Маршрут | Описание |
|---|---|---|
| `POST` | `/api/setup` | Настройка улья при первом запуске |
| `POST` | `/api/dispatch` | Отправляет цель рою (без состояния) |
| `GET` | `/api/tools` | Список всех доступных инструментов |
| `GET` | `/ws` | WebSocket-соединение для событий в реальном времени |

#### Сессии

| Метод | Маршрут | Описание |
|---|---|---|
| `GET` | `/api/sessions` | Список всех сессий |
| `POST` | `/api/sessions` | Создать сессию |
| `GET` | `/api/sessions/:id` | Получить сессию с агентами |
| `DELETE` | `/api/sessions/:id` | Удалить сессию (каскад) |
| `POST` | `/api/sessions/:id/dispatch` | Запустить рабочий процесс для сессии |
| `GET` | `/api/sessions/:id/agents` | Список агентов сессии |
| `POST` | `/api/sessions/:id/documents` | Загрузить и проиндексировать документ |

#### Постоянные ульи (Colmeias)

Ульи — именованные постоянные сущности. В отличие от сессий, улей может получать **несколько сообщений с течением времени**, сохраняя историю разговоров как контекст для каждой новой отправки. Агенты могут быть **заранее определены пользователем** (с настраиваемыми промптами и инструментами, только при `queen_managed=false`) или **собраны Королевой автоматически** (`queen_managed=true`). Попытка добавить предопределённых агентов в улей с `queen_managed=true` вернёт `409 Conflict`.

| Метод | Маршрут | Описание |
|---|---|---|
| `GET` | `/api/colmeias` | Список всех ульев |
| `POST` | `/api/colmeias` | Создать улей (`queen_managed: true/false`) |
| `GET` | `/api/colmeias/:id` | Получить улей с агентами |
| `PUT` | `/api/colmeias/:id` | Обновить улей |
| `DELETE` | `/api/colmeias/:id` | Удалить улей (каскад) |
| `POST` | `/api/colmeias/:id/dispatch` | Отправить сообщение улью |
| `GET` | `/api/colmeias/:id/historico` | История разговоров |
| `GET` | `/api/colmeias/:id/agentes` | Список агентов улья |
| `POST` | `/api/colmeias/:id/agentes` | Добавить предопределённого агента (требуется `queen_managed=false`) |
| `GET` | `/api/colmeias/:id/agentes/:agentId` | Получить агента по ID |
| `PUT` | `/api/colmeias/:id/agentes/:agentId` | Редактировать агента (промпт, инструменты) |
| `DELETE` | `/api/colmeias/:id/agentes/:agentId` | Удалить агента из улья |

#### Вебхуки

| Метод    | Маршрут                 | Описание                                                    |
| -------- | ----------------------- | ----------------------------------------------------------- |
| `POST`   | `/api/webhooks/:slug`   | **Публичный** — запускает workflow связанного улья          |
| `GET`    | `/api/webhooks`         | Список всех вебхуков                                        |
| `POST`   | `/api/webhooks`         | Создать вебхук                                              |
| `GET`    | `/api/webhooks/:id`     | Получить вебхук по ID                                       |
| `PUT`    | `/api/webhooks/:id`     | Обновить вебхук                                             |
| `DELETE` | `/api/webhooks/:id`     | Удалить вебхук                                              |

**Пример — создание улья с агентами, определёнными пользователем:**

```bash
# 1. Создать улей
curl -X POST http://localhost:8080/api/colmeias \
  -H "Content-Type: application/json" \
  -d '{"name": "Исследовательский улей", "queen_managed": false}'

# 2. Добавить агента с настраиваемым промптом
curl -X POST http://localhost:8080/api/colmeias/{id}/agentes \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Веб-исследователь",
    "system_prompt": "Вы специалист по исследованиям. Используйте web_search для сбора актуальной информации.",
    "allowed_tools": ["web_search", "search_memory", "store_memory"]
  }'

# 3. Отправить первое сообщение
curl -X POST http://localhost:8080/api/colmeias/{id}/dispatch \
  -H "Content-Type: application/json" \
  -d '{"goal": "Найдите главные новости об ИИ за эту неделю"}'

# 4. Отправить следующее сообщение (история автоматически добавляется как контекст)
curl -X POST http://localhost:8080/api/colmeias/{id}/dispatch \
  -H "Content-Type: application/json" \
  -d '{"goal": "На основе предыдущего исследования напишите резюме для руководства"}'
```

#### `POST /api/dispatch`

```json
// Запрос
{ "goal": "Создай файл Go, который складывает два числа", "group_id": "enxame-alfa" }

// Ответ 202
{ "message": "Mission dispatched to the swarm. Follow progress via WebSocket." }
```

### События WebSocket (`/ws`)

#### Сервер → Фронтенд

| `type` | Когда срабатывает | Важные поля |
|---|---|---|
| `status` | Сообщения о прогрессе от Королевы | `message` |
| `agent_change` | Специалист берёт управление конвейером | `agent` |
| `tool_start` | Инструмент готовится к выполнению | `agent`, `tool`, `args` |
| `approval_request` | ИИ хочет использовать ограниченный инструмент | `id`, `tool`, `args` |
| `result` | Итоговый отчёт рабочего процесса | `message` |
| `error` | Ошибка или тайм-аут | `message` |

#### Фронтенд → Сервер

```json
{ "type": "approve", "id": "req-1712345678901", "approved": true }
{ "type": "approve", "id": "req-1712345678901", "approved": false }
```
