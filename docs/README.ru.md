## 📖 Почему «Jandaira»?

**Jandaíra** (*Melipona subnitida*) — безжальная пчела, эндемичная для биома Каатинга в Бразилии. Маленькая, стойкая и исключительно кооперативная — она не нуждается в централизованном лидере, чтобы построить функциональный улей. Каждая рабочая пчела знает свою роль, автономно выполняет задачу и возвращает результат коллективу.

Именно эта архитектурная модель реализована в данном проекте:

- **Королева (`Queen`)** не выполняет задачи — она оркестрирует, проверяет политики и обеспечивает безопасность.
- **Специалисты (`Specialists`)** — лёгкие агенты с ограниченным набором инструментов, работающие в изолированных силосах.
- **Нектар (Nectar)** — метафора для бюджета токенов: каждый агент потребляет нектар; когда он заканчивается, улей останавливается.
- **Соты (`Honeycomb`)** — общая векторная память — коллективные знания, сохраняющиеся между миссиями, хранящиеся в ChromaDB.
- **Пасечник (Beekeeper)** — человек в контуре: он может одобрить или заблокировать любое действие ИИ до его выполнения.

## ⚡ Преимущества перед NanoClaw

| Характеристика | NanoClaw (Python) | Jandaira (Go) |
|---|---|---|
| **Язык** | Python | Go 1.22+ |
| **Параллелизм** | `asyncio` / потоки | Нативные горутины + каналы |
| **Изоляция агентов** | Docker-контейнеры | Wasm через `wazero` (без Docker) |
| **IPC-коммуникация** | JSON на диске / Redis | Типизированная общая память |
| **Шифрование между агентами** | ❌ Отсутствует | ✅ AES-GCM между каждой передачей |
| **Человек в контуре** | Опционально / внешний | ✅ Встроенный: режим пасечника через WebSocket |
| **Бюджет токенов** | Ручной | ✅ Автоматический `NectarUsage` на рой |
| **Векторная память** | Pinecone / внешняя | ✅ ChromaDB via Docker |
| **Интерфейс** | Отсутствует | ✅ REST API + WebSocket |
| **IPC-задержка** | Высокая (дисковый/сетевой I/O) | Минимальная (память) |

### Почему Go превосходит Python здесь?

1. **Горутины дешевле потоков** — запуск 100 одновременных агентов стоит ничтожно мало по сравнению с Python `asyncio` или `threading`.
2. **Статический бинарный файл** — нулевые зависимости во время выполнения. `go build` генерирует исполняемый файл, который работает на любом Linux без установки чего-либо.
3. **Нет GIL** — в Python есть Global Interpreter Lock; Go по-настоящему параллелизует работу на нескольких ядрах.
4. **`wazero` — это 100% Go** — Wasm-среда выполнения не требует CGo, Docker или внешних систем. Агент работает в песочнице внутри того же процесса.

## 🚀 Руководство по использованию

### Предварительные требования

```bash
# Go 1.22 или выше
go version

# Docker (для ChromaDB)
docker --version

# Ключ OpenAI API
export OPENAI_API_KEY="sk-..."
```

### Запуск ChromaDB

```bash
# Напрямую через Docker
docker run -d --name chroma -p 8000:8000 chromadb/chroma:latest

# Или с помощью docker-compose проекта
docker compose up -d
```

По умолчанию сервер подключается к `http://localhost:8000`. Для другого адреса:

```bash
export CHROMA_URL="http://my-chroma:8000"
```

### Установка

#### Вариант 1 — Сборка из исходного кода

```bash
git clone https://github.com/damiaoterto/jandaira.git
cd jandaira

# Загрузить зависимости
go mod tidy

# Собрать API-сервер
go build -o jandaira-api ./cmd/api/
```

#### Вариант 2 — Запустить напрямую

```bash
go run ./cmd/api/main.go --port 8080
```

### Запуск улья

```bash
./jandaira-api --port 8080
```

Сервер будет доступен по адресу `http://localhost:8080`. Мониторинг событий в реальном времени через WebSocket: `ws://localhost:8080/ws`.

### Пример: создание и тестирование файла Go

1. Отправьте цель через `POST /api/dispatch`:

   ```bash
   curl -X POST http://localhost:8080/api/dispatch \
     -H "Content-Type: application/json" \
     -d '{"goal": "Создай файл Go с именем sum.go, который складывает два числа", "group_id": "enxame-alfa"}'
   ```

2. Королева распределяет задачу по конвейеру специалистов:
   - **Wasm-разработчик** → пишет `sum.go` с помощью `write_file`
   - **Аудитор качества** → выполняет код с помощью `execute_code` и генерирует отчёт

3. Отслеживайте прогресс через WebSocket:

   ```json
   { "type": "agent_change", "agent": "Wasm-разработчик" }
   { "type": "tool_start",   "agent": "Wasm-разработчик", "tool": "write_file", "args": "{...}" }
   { "type": "result",       "message": "# Итоговый отчёт\n..." }
   ```

4. Если `RequiresApproval: true`, **режим пасечника** активируется. Сервер отправляет `approval_request` через WebSocket и ждёт ответа:

   ```json
   // Сервер отправляет:
   { "type": "approval_request", "id": "req-1712345678901", "tool": "write_file", "args": "{...}" }

   // Клиент отвечает:
   { "type": "approve", "id": "req-1712345678901", "approved": true }
   ```

5. В конце результат сохраняется в векторную память ChromaDB для будущего использования.

### Настройка собственного роя

Отредактируйте `cmd/api/main.go`, чтобы определить политику роя:

```go
queen.RegisterSwarm("my-swarm", swarm.Policy{
    MaxNectar:        50000,  // Бюджет токенов
    Isolate:          true,   // Изолированный контекст на группу
    RequiresApproval: true,   // Режим пасечника (HIL)
})
```

### Доступные инструменты

| Инструмент | Описание |
|---|---|
| `list_directory` | Перечисляет файлы и папки в директории |
| `read_file` | Читает содержимое файла |
| `write_file` | Создаёт или перезаписывает файл |
| `execute_code` | Выполняет код в изолированной Wasm-песочнице |
| `search_memory` | Семантический поиск в векторной памяти улья (ChromaDB) |
| `store_memory` | Сохраняет знания в векторную память |

## 🔐 Безопасность

Каждая «эстафетная передача» между специалистами **зашифрована с помощью AES-GCM**:

1. В начале каждого рабочего процесса генерируется эфемерный сессионный ключ
2. Накопленный контекст **шифруется перед отправкой** следующему специалисту
3. Специалист получает зашифрованную полезную нагрузку, расшифровывает, обрабатывает и **повторно шифрует** свой ответ
4. Никакой контекст не передаётся в открытом виде между агентами

Это имитирует защищённый IPC-канал, где даже если один агент скомпрометирован, он не может прочитать историю других агентов в конвейере.

## 🌐 Справочник API

Запустите HTTP-сервер командой `./jandaira-api --port 8080`. Доступны следующие маршруты:

### REST-маршруты

| Метод | Маршрут | Описание |
|---|---|---|
| `POST` | `/api/dispatch` | Отправляет цель рою для выполнения |
| `GET` | `/api/tools` | Возвращает список всех доступных инструментов |
| `GET` | `/api/agents` | Возвращает список специалистов в рабочем процессе |
| `GET` | `/ws` | Открывает WebSocket-соединение для событий в реальном времени |

#### `POST /api/dispatch`

```json
// Запрос
{ "goal": "Создай файл Go, который складывает два числа", "group_id": "enxame-alfa" }

// Ответ 202
{ "message": "Mission dispatched to the swarm. Follow progress via WebSocket." }
```

---

### События WebSocket (`/ws`)

#### Сервер → Фронтенд

| `type` | Когда срабатывает | Важные поля |
|---|---|---|
| `status` | Сообщения о прогрессе от Королевы | `message` |
| `agent_change` | Специалист берёт управление конвейером | `agent` |
| `tool_start` | Инструмент готовится к выполнению | `agent`, `tool`, `args` |
| `approval_request` | ИИ хочет использовать ограниченный инструмент | `id`, `tool`, `args` |
| `result` | Итоговый отчёт рабочего процесса | `message` |
| `error` | Ошибка или тайм-аут | `message` |

#### Фронтенд → Сервер

```json
{ "type": "approve", "id": "req-1712345678901", "approved": true }
{ "type": "approve", "id": "req-1712345678901", "approved": false }
```
